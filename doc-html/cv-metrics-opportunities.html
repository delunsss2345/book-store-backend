Top 3 nên làm trước

Ba hạng mục này cho số đẹp nhất, đáng tin nhất, và dễ chứng minh trong phỏng vấn.

1. RBAC cache key mismatch

Fix nhỏ, bằng chứng rất rõ: đọc role_per nhưng ghi role_user. Dễ benchmark trước/sau bằng DB query count và cache hit-rate.

  • Con số đẹp: DB permission reads/request, hit-rate, p95 latency.
  • Đáng tin: có thể demo Redis keys và Prisma query log ngay trong phỏng vấn.

2. Payment webhook idempotency

Đây là câu chuyện backend mạnh nhất vì liên quan tiền, retry callback, transaction và duplicate processing.

  • Con số đẹp: 100 duplicate callbacks → 1 transition, 0 duplicate side effects.
  • Đáng tin: có thể có integration test bắn song song cùng providerEventId.

3. Semantic search cache hoàn chỉnh

Code hiện đã có cache.get nhưng thiếu cache.set trong searchBooks. Sau khi fix, số giảm external calls thường rất dễ nhìn.

  • Con số đẹp: Gemini/Pinecone calls giảm, cost/query giảm, p95 latency giảm.
  • Đáng tin: mock/counter service là đủ chứng minh trước khi có billing thật.

Những con số dễ bị nghi ngờ fake

Các số dưới đây vẫn dùng được, nhưng phải có bằng chứng đi kèm nếu đưa vào CV.

"Giảm latency X%"

Dễ bị hỏi vì latency phụ thuộc máy, DB size, network, Redis, warm/cold cache.

  • Chứng minh bằng script benchmark commit vào repo.
  • Lưu output thô: p50/p95, concurrency, dataset size, env.
  • Demo chạy lại được bằng Docker Compose hoặc staging DB.

"0 overselling / exactly once"

Đây là số mạnh nhưng phải là concurrency test, không phải test tuần tự.

  • Commit test dùng Promise.all hoặc k6 với cùng variant stock thấp.
  • Log SQL/query sau test: stock, reserved, orders, order_items.
  • Giữ seed data và câu lệnh chạy lại.

"Tăng hit-rate recommendation"

Rất dễ bị nghi nếu không nói rõ offline evaluation protocol.

  • Tách train/test theo thời gian, không random leakage.
  • So với baseline popularity hoặc newest books.
  • Report hit-rate@K, coverage và số user/event dùng để đo.

"Giảm cost AI/search"

Nếu chưa có billing thật, chỉ nên ghi số calls avoided hoặc estimated cost.

  • Instrument counter cho Gemini/Pinecone calls.
  • Ghi rõ workload: số query lặp, unique query, cache TTL.
  • Nếu dùng estimate, tách khỏi số billing thật.

"Coverage tăng lên C%"

Coverage tự thân không chứng minh chất lượng nếu test chỉ cover mapper/dto.

  • Ưu tiên test payment webhook, RBAC guard, inventory race, auth token.
  • Đính kèm coverage report và danh sách critical paths covered.
  • Nhắc rõ baseline hiện tại: Jest root src không có unit specs.

"Giảm ảnh X%"

Phụ thuộc tập ảnh. Không được lấy một ảnh đẹp nhất làm đại diện.

  • Dùng ít nhất 30-100 ảnh đại diện: JPG/PNG/WebP, cover lớn/nhỏ.
  • Báo median, p90, min/max, không chỉ average.
  • Lưu manifest trước/sau bytes và thời gian optimize.

Dữ kiện còn thiếu

Cần các dữ kiện này để biến cơ hội thành con số CV thật, có thể bảo vệ khi phỏng vấn.

Dữ liệu chạy thật

  • DB snapshot hoặc seed lớn: users, books, variants, events, orders, email_outbox, webhook_inbox.
  • Redis/BullMQ metrics nếu có: queue wait time, retries, failed jobs.
  • Traffic/log thực tế: request rate, slow endpoints, payment polling pattern.

Môi trường đo

  • Docker Compose/staging chạy được đủ MariaDB, Redis, app.
  • Thông số máy đo: CPU/RAM, Node version, DB version, network local/staging.
  • API keys hoặc mock cho Gemini/Pinecone/R2 để tách latency nội bộ và external.

Benchmark scripts

  • Load test: autocannon/k6 cho catalog, search, webhook, order status.
  • Concurrency test: checkout stock thấp, duplicate webhook, refresh/logout token.
  • Offline evaluator: recommendation hit-rate@K và search relevance test set.

Observability

  • Structured log có request id, route, status, duration, user/session id khi an toàn.
  • Prisma query timing/query count per request trong benchmark.
  • Metrics counters: cache hit/miss, external API calls, webhook duplicate, queue retries.

Commit history

  • Commit trước fix, commit benchmark baseline, commit implementation, commit after benchmark.
  • Không sửa benchmark sau khi có số đẹp nếu không có lý do rõ.
  • Giữ raw output trong docs/benchmarks/ hoặc CI artifacts.

Ranh giới chưa được bịa

  • Chưa có số production latency, error rate, cost AI thật.
  • Chưa có coverage report hợp lệ vì hiện tại không có unit specs trong src.
  • Chưa có dataset để claim recommendation/search quality.
