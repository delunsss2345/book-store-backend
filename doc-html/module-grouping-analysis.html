Semantic search and AI recommendation stack. Three tightly coupled modules: gemini generates embeddings, pinecone stores/queries vectors, search orchestrates the full pipeline on top of the catalog.

All three modules form a chain: gemini → pinecone → search. search also imports catalog directly for hybrid lexical+vector queries.

Group 6

Messaging

Transactional email pipeline built on the outbox pattern with BullMQ workers. email-outbox is the persistence layer; mail is the SMTP sender; jobs orchestrates the queue between them.

Messaging 3 modules

The pattern is correct: domain modules write to email-outbox (persist then fire), jobs picks up via BullMQ, and mail delivers. No domain module imports mail directly — isolation is maintained.

email-outbox
Transactional outbox — stores email payloads in DB before async delivery. Consumed broadly: auth, order, verification-code.

mail
Nodemailer SMTP adapter. Standalone leaf — only consumed by jobs. Domain modules never import mail directly.
—
jobs
BullMQ email queue — producers, processors, and Redis plumbing. Bridges email-outbox → mail. Also handles verification code delivery.

Issues

Architecture smells

Three verified issues found by tracing imports verbatim from source. Ranked by severity.

🔴
order leaks into auth internals
OrderModule imports AuthModule at line 28, which correctly exposes AuthService. But it also declares AuthRepository directly in its own providers array at line 48 — bypassing AuthModule's encapsulation boundary and duplicating a provider that should be private to the auth domain. If AuthRepository ever changes its constructor signature, OrderModule silently breaks.
order/order.module.ts · line 12 (import) · line 48 (provider)
🔴
admin is a god-module
AdminModule registers 8 controllers, 10 providers, and 11 exports in a single module. It inlines AdminBookService, AdminOrderService, AdminUserService, AdminCategoryService, and AdminBookVariantsService — all of which re-implement data-access logic already owned by their respective domain modules (book/catalog, order, user, category). Any bug fix in the domain model must be applied in two places.
admin/admin.module.ts · lines 36–45 (controllers) · lines 48–57 (providers)
🟡
R2ServiceService injected directly in two modules
Both UploadsModule and BookAssetModule inject R2ServiceService directly into their providers array instead of importing R2ServiceModule. This bypasses the DI module boundary: if R2ServiceService gains a required constructor dependency, both callers break silently at runtime. The fix is to import R2ServiceModule and let NestJS resolve the dependency through the module graph.
uploads/uploads.module.ts · providers[1] · book/asset/book-asset.module.ts · providers[2]
