Issue · Data integrity

Inventory is split across two contradictory definitions — available is never written, and order expiry inflates stock

Severity: Critical bug data-integrity money-path module: cart · order · book/variant model: BookVariant
Found via codebase + schema audit Branch dev Date 2026-06-21 Confidence verified (read on source)
TL;DR Two definitions Evidence Reproduction Impact Root cause Proposed fix Migration Runner-ups
TL;DR

BookVariant carries three inventory fields — stock, reserved, and a materialized available (intended to mean stock − reserved). The cart path gates add-to-cart on the available column; the order path computes stock − reserved inline and ignores the column entirely. No code anywhere writes available, so it is frozen at its DB default of 0.

Two consequences, both on the money path, both silent: (A) every variant created through the normal admin / stock-import flow is un-addable to any cart — the store looks entirely sold out while inventory is full; and (B) order expiry runs stock += qty for stock that was never debited at reserve time, so stock drifts upward on every abandoned order → eventual oversell.

01Two definitions of "purchasable", never reconciled

There is no single source of truth for "how many units can a customer buy right now." Two subsystems answer the question differently, and a third writer (expiry) mutates a field neither of them expects:

Because the available column is never maintained, the cart and storefront permanently disagree with the order path. The column is a materialized view that has no writer.

02Evidence

The schema declares three fields; available defaults to 0

stock and reserved are nullable; available is non-null with default 0. That default is the value every row keeps forever (see below).

Add-to-cart gates only on available

The quantity-increase path applies the same clamp — Math.max(0, stock?.available ?? 0) at src/modules/cart/service/cart.service.ts:358 — so it, too, collapses to 0.

Nothing ever writes available

A whole-tree search for the identifier across src/ returns reads only — selects, a DTO field, and the divergent order formula. There is not a single assignment:

The inventory-mutating repository confirms it: stock import updates stock/costPrice/price and never touches available (src/modules/book/variant/repository/bookVariant.repository.ts:21–41), and the seed leaves the assignment commented out (prisma/seed.ts:1165, :1178).

The order path computes a different number and reserves by bumping reserved only

The intended model is clearly available = stock − reserved (the order path even names the local variable available). On reserve, stock is deliberately not debited — units only leave at fulfillment. So expiry must return the hold (reserved −qty) and leave stock alone. Instead it adds qty back to stock, inventing units.

03Reproduction

Failure A — the purchase funnel is dead on arrival

Setup: admin (or stock-import) creates variant #42 with stock = 100. Nothing sets available, so it stays 0; reserved = 0.

  1. Customer opens the product and clicks Add to cart, quantity 1.
  2. cart-item.service.ts:31 evaluates available (0) < 1 → throws Error("Not enough stock").
  3. The variant — with 100 real units — cannot be added to any cart by anyone, ever. The whole catalog appears sold out. The same clamp blocks quantity increases.

Failure B — stock inflation → oversell

Setup: a cart does exist (legacy row, or a seed that set available). Variant #42: stock = 5, reserved = 0.

  1. Customer places an order for 5 → reserve runs: reserved = 5, stock = 5 (order check stock − reserved = 0… passes at the boundary for qty ≤ 5).
  2. Customer never pays; the cleanup job expires the order.
  3. clearOrder runs: stock = 5 + 5 = 10, reserved = 0.
  4. Inventory now claims 10 units for a variant that physically has 5. Repeat across abandoned orders and stock drifts upward without bound.

Cart abandonment is typically 60–80%, so Failure B accrues on the majority of orders that reach reservation.

04Impact at scale

DimensionConsequence
RevenueFailure A is a complete checkout outage for any variant created through the normal flow — 0 add-to-cart conversions, surfaced only as a generic per-request Error("Not enough stock") that no monitor flags as a config fault.
Inventory accuracyFailure B corrupts the stock ledger proportionally to abandonment volume. Once stock overstates reality, paid orders are accepted that cannot be fulfilled.
DownstreamOversell → cancellations, refunds, chargebacks, support load; cost_price-derived inventory valuation drifts, polluting any reporting built on it.
ObservabilityBoth failures are silent — no exception distinguishes "misconfigured inventory" from "genuinely out of stock."

05Root cause

There are two independent, never-reconciled definitions of purchasable quantity:

  • a materialized available column (intended stock − reserved), trusted by cart + storefront; and
  • an on-the-fly stock − reserved computed by order.

No writer maintains the column, so the two permanently disagree. Compounding it, the reserve step mutates only reserved, while the expiry path mutates stock — a field that the reserve step never changed — so the restock arithmetic is internally inconsistent with the "hold = stock→reserved" model the rest of the system assumes.

06Proposed fix

Pick one inventory model and make every path obey it. Recommended: keep available as the single source of truth for purchasability and maintain it transactionally next to reserved.

1 · Schema — make the invariant enforceable

Non-null stock/reserved removes the stock!/reserved! non-null assertions and the !v.stock bug that conflates 0 with null.

2 · Maintain available atomically on every transition (inside the existing tx)

Success (paid)−qty−qty—

3 · Make reserve a guarded, atomic write (also fixes the latent oversell race)

The current check-then-increment at order.service.ts:138/:192 is a read-modify-write across two statements — a TOCTOU window under concurrency. Collapse it into one conditional update:

This removes the race without needing SELECT … FOR UPDATE, because the available >= qty predicate and the decrement are evaluated atomically by the database.

4 · Unify the readers

Delete the divergent stock − reserved computation in order.service.ts and have the order check read the same available column the cart does. One number, one place.

Trade-off

Maintaining a materialized available means one more field to keep in sync on every transition (more places to get wrong), versus dropping available and computing stock − reserved everywhere (one fewer column, simpler invariant — but every read site changes and you lose an indexable "in-stock" flag). Since the cart and storefront already depend on the column, maintaining it is the smaller, lower-risk change. Either way the non-negotiable is one model, enforced in one place.

07Migration plan

  1. Normalize existing nulls before adding NOT NULL/CHECK: set stock/reserved nulls to 0, and clamp any already-negative values produced by the current buggy expiry path.
  2. Backfill the column in the same release as the code change:
  3. Without the backfill, every existing row stays at available = 0 and Failure A persists after deploy.
  4. Ship backfill + maintenance logic together (ideally with writes quiesced) to avoid a window where in-flight reserves race the backfill.
