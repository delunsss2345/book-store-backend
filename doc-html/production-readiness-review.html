Bookstore Backend:
What breaks first, and what it will cost.

A critical review of the NestJS 11 / Prisma 7 / MariaDB backend — 527 files, ~50 modules, 40 tables — read against one goal: minimum infrastructure cost while staying smooth under load, horizontally scalable, maintainable, and reasonably fast to build on. The foundations are good (clean layering, transactional outbox, idempotent webhooks, snapshots). This document is about the 15 things that will bite as traffic and data grow — ranked by impact × effort, each with a concrete fix and its trade-off.

Bottom line. Nothing here is a rewrite. The stack is appropriate and the patterns are above-average for the size. But three choices actively block the cheapest scaling path (add small instances behind a load balancer): the in-process cache tier, the per-request DB auth check, and an OLTP database that also stores the analytics firehose. Fix those first — they are the difference between "scale by adding $10 instances" and "scale by paying for one ever-bigger one."

01

Cost & scaling — the primary goal

The stated objective is minimum infra cost with easy horizontal/vertical scaling. Three design choices currently force the expensive direction: scaling up one big box instead of out across cheap small ones.

Critical

In-process L1 cache blocks horizontal scaling

C1
src/config/redis.config.ts:71-77 · two-tier store: CacheableMemory(lruSize 5000, ttl 60s) → Redis
Problem
The cache is a two-tier stack: an in-process memory store in front of Redis. Each Node instance keeps its own L1. Any invalidation (catalog edit, permission change, order status) only clears the local process — peers serve stale data until their 60s TTL lapses.
Impact at scale
The cheapest way to handle traffic is N small instances behind a load balancer. With per-process L1, those instances disagree: a price edit shows on the node that served the write and is stale everywhere else. To stay correct you’re pushed toward one large vertical instance — the opposite of the cost goal — or you ship visible inconsistency.
Proposal
Make Redis the single cache tier (drop the memory Keyv), or keep L1 only for truly immutable data (enum/i18n lookups) with explicit pub/sub invalidation. Redis-only keeps every instance consistent and lets you scale out on $5–10 nodes.
Trade-off
Every cache read becomes a Redis round-trip (~0.3–1ms) instead of an in-process hit. For this workload that is negligible and worth the correctness; revisit L1 only if a profiler shows Redis as the bottleneck.
Critical

A database round-trip on every authenticated request

C3
auth.guard.ts:46 → revoked-token.repository.ts:23 — findUnique per request · no cleanup cron
Problem
Every guarded request verifies the JWT (cheap, in-memory) and issues revokedToken.findUnique() to MariaDB to check the blacklist. The deep-dive bills RBAC as “authorization without DB round-trips,” but authentication itself hits the DB on 100% of requests. The blacklist table also has no cleanup job — it grows forever (one row per logout), so the hottest query in the system runs against an ever-larger table.
Impact at scale
DB connections and QPS scale linearly with traffic on the one query you can’t cache away, forcing a bigger (pricier) DB instance and shrinking headroom for checkout. Unbounded revoked_tokens inflates backups and slowly raises p99 auth latency.
Proposal
Move revocation to Redis: store revoked:{hash} with a TTL equal to the token’s remaining life — the entry self-expires, so no cleanup job and no growth. Auth becomes a sub-millisecond Redis EXISTS. Keep access-token TTLs short (5–15 min) so the blacklist stays tiny.
Trade-off
Revocation now depends on Redis availability — but Redis is already required for cache + queue, so it adds no new dependency. If absolute durability of revocations matters, keep a DB write-through and read from Redis.
High

One database holds OLTP and the analytics firehose

H1
prisma/models/engagement.prisma · user_events / audit_logs · login_attempts (user.prisma)
Problem
user_events records a row for every view, search, add-to-cart and payment. It is append-only, unbounded, and lives in the same MariaDB as orders and inventory, with no TTL, partitioning, or archival. It will become the largest table by an order of magnitude, and the recommendation engine scans it on the primary.
Impact at scale
The transactional DB you pay to keep fast, replicated, and backed-up is dominated by analytics rows that checkout never needs. Backups slow, storage cost climbs, and recommendation reads compete with order writes for buffer pool and IO.
Proposal
Separate the firehose: range-partition user_events by month with a TTL/drop-old job (cheapest), or stream events to an append-only store (ClickHouse / object storage + Athena) and keep MariaDB for transactions. Same treatment for audit_logs and login_attempts.
Trade-off
A second store adds operational surface. Start with partition + TTL on the existing DB (near-zero new infra) and only externalize when event volume justifies it — staged by cost, not bought up-front.
Cost lever, not a bug: semantic search runs through Pinecone (a paid managed vector DB) plus a Gemini embedding per query. For a catalog likely under ~100k titles that is a fixed monthly vendor bill for a feature most queries don’t need. See M4 — gate vector search behind cheap keyword search and consider collapsing vectors into the DB.
02

Correctness & money

Where mistakes cost real money or destroy records. These are not style issues — they change data.

Critical

Stock reservation has a check-then-act race → oversell

C4
src/modules/order/order.service.ts:86-158 (processOrderItems)
Problem
Availability is read via catalogRepository (outside the transaction’s locks), checked in JS as stock - reserved, and only later does tx.bookVariant.update({ reserved: { increment } }) run. Two concurrent checkouts for the last copy both read “1 available,” both pass, both reserve → oversell. The increment is atomic; the guard is not.
Impact at scale
Under any concurrency on hot/limited titles you sell stock you don’t have — refunds, support load, and broken trust on exactly the launches that drive traffic. stock and reserved are also nullable, and the code force-unwraps (variant.stock!), risking a runtime error if either is null.
Proposal
Make the reservation a single conditional write the DB enforces: UPDATE book_variants SET reserved = reserved + :q WHERE id = :id AND (stock - reserved) >= :q and treat 0 rows affected as out-of-stock. That collapses check-and-act into one atomic statement — no lock juggling, no oversell.
Trade-off
Slightly less ergonomic than Prisma’s typed update (needs $executeRaw or updateMany + affected-count check), and contended rows serialize briefly. That serialization is the point — it’s correctness on the one row that matters.
Critical

Deleting an order cascade-deletes its payment records

C2
prisma/models/commerce.prisma:165 — PaymentTransaction.order onDelete: Cascade
Problem
A PaymentTransaction is cascade-deleted with its Order. Orders have no soft-delete, so a hard delete (cleanup, admin action, GDPR erasure) silently destroys the financial ledger — including the raw gateway payloads kept for reconciliation.
Impact at scale
Loss of payment history is an accounting and compliance hazard: you can’t reconcile, dispute a chargeback, or prove a refund. It’s the one record class that must outlive everything around it.
Proposal
Change the relation to onDelete: Restrict (or SetNull + retain). Never hard-delete orders with payments — soft-delete them. Treat payment_transactions and webhook_inbox as an immutable ledger.
Trade-off
Deleting an order now requires handling its payments explicitly (archive/anonymize) rather than a free cascade — which is exactly the friction you want guarding financial data.
High

Order money & status columns are nullable

H3
prisma/models/commerce.prisma:28-33 — status, paymentStatus, subtotal, totalAmount all ?
Problem
An Order can exist with NULL status, payment status, subtotal and total. Invariants that should be guaranteed by the schema are left to application code, so every report and admin query must defend against NULL money.
Impact at scale
Revenue sums silently skip NULL rows; dashboards under-count; a partial write leaves an order in an unrepresentable state. Bugs surface as “the numbers don’t add up,” which are slow and expensive to chase.
Proposal
Make them NOT NULL with defaults (status = PENDING_PAYMENT, amounts default 0) via a backfill migration. Add DB CHECK constraints (total >= 0, rating BETWEEN 1 AND 5) — MariaDB enforces them now.
Trade-off
A backfill + migration is required, and any code relying on NULL as “not set yet” needs an explicit initial state — a one-time cost for permanent integrity.
High

paymentUrl VARCHAR(100) will truncate real gateway URLs

H4
commerce.prisma:157 — PaymentTransaction.paymentUrl @db.VarChar(100) (vs PaymentIntent.paymentUrl VarChar(200))
Problem
Gateway redirect/QR URLs (VNPAY, Stripe, MoMo) routinely exceed 100 characters with their signed query strings. The same concept is 200 on PaymentIntent and 100 here — inconsistent, and 100 is too short.
Impact at scale
On a real gateway switch this is a hard insert error (or silent truncation) in the payment path — failures land exactly where you can least afford them.
Proposal
Widen to VarChar(1000) (or Text) and standardize URL lengths across payment tables. Trivial migration.
Trade-off
None worth noting; VARCHAR length costs nothing until used.

Database design review

40 tables across 8 well-organized schema files. The modeling instincts are strong — snapshots, audit log, a documented onDelete matrix, composite keys. The gaps are in indexing for real query shapes, a few constraints, and growth planning.

Indexing strategy — the biggest gap

InnoDB auto-indexes single FK columns, so simple lookups are covered. What’s missing are composite indexes for the actual access patterns and a few uniqueness constraints the app only enforces in code.

And with no @@unique([userId, bookVariantId]), the “already reviewed?” check is a racy app-level guard — two fast requests create duplicate reviews.

04

Module architecture & structure

The controller → service → repository layering is clean and consistent — a real strength. The problem is granularity: ~50 modules where many wrap a single join table, while a few services have swollen into god-files.

High

One-table-per-module fragmentation vs. god-files

H5
Problem
Two opposite smells at once. Anemic sprawl: order-item, cart-item, order-address, book-author, user-role, role-permission are standalone modules for what are really parts of an aggregate. God-files: catalog.repository.ts (832 lines, 19 query methods), order.service.ts (567), catalog.service.ts (547), admin-book.service.ts (513).
Impact at scale
Every feature touches many tiny modules (wiring tax, hard to trace a flow), while the few files that matter most are the hardest to change safely. New engineers pay both costs. The money path (order.service) even uses any for tx/cart, discarding type safety where it’s most valuable.
Proposal
Reorganize around bounded contexts (aggregates), not tables. Collapse join-table modules into their owning aggregate; split god-files by use-case (read-model vs. write-model). See the map below.
Trade-off
A refactor with no user-visible payoff, so it competes with features. Do it incrementally — fold one join-module per PR, and carve a CatalogReadModel out of the catalog repository first, since that file changes most.

05

Security, API & developer experience

Security

High

Payment webhook isn’t signature-verified

S1
src/modules/hooks/* · only an API-key guard + idempotency today
Problem
The Sepay webhook is protected by a shared API key and idempotency, but the request body’s authenticity isn’t cryptographically verified. Anyone who learns the endpoint + key (or replays a captured call with new content) can assert a payment.
Impact at scale
A forged “payment success” marks an order PAID and ships goods for free. This is the highest-value attack surface in the system.
Proposal
Verify the provider HMAC/checksum over the raw body before processing; reject on mismatch. Combine with the existing idempotency + amount check.
Trade-off
Must capture the raw body (bypass the JSON parser for that route) to hash correctly — a small middleware change.
IssueDetail & fix
high Stale permissionsPermission cache writes key role_user:{id}:perms but reads role_per:{id}:perms — the cache never hits (so authz silently falls back to DB), and there’s no invalidation on grant/revoke. Unify the key, add active invalidation, drop the leftover console.log(userPerms). also a cost fix
med TOTP secret at restusers.totpSecret VARCHAR(255) stored plaintext. Encrypt at rest (app-level envelope or DB encryption) before MFA ships.
med Refresh rotationRefresh tokens are hashed (good) but not rotated with reuse-detection. Rotate on each use and revoke the session family on replay.
low Enumerable IDsSequential BigInt order/user IDs are guessable; ensure every by-ID endpoint enforces ownership (defense in depth against IDOR).

One-paragraph verdict. This is a well-built application carrying a few decisions that quietly force the expensive scaling path and put financial data at risk. Spend one sprint on the P0 list — a handful of migrations and moving two checks to Redis — and you convert “scale by buying a bigger box” into “scale by adding cheap instances,” with the payment ledger and inventory made trustworthy. Everything after that is steady, incremental hardening, not heroics.
