Three ways to authorize a request

Prompt The app guards every protected route with a global PermissionsGuard. Show me three different ways to resolve a user's permission set on each request, with the tradeoffs for each — and tell me which one this codebase should keep.

01Join straight to the database

Resolve role → permission rows from Prisma on every guarded request, no cache.

// permission.guard.ts — naive form
const { user } = ctx.switchToHttp().getRequest();
const roleIds =
  await userRoleService.getRoleIdsByUserId(user.sub);

const perms = await Promise.all(
  roleIds.map((id) =>
    rolePermissionService.getByRoleId(id)),
);

if (!new Set(perms.flat()).has(required))
  throw new ForbiddenException();
Pro
Con
Always fresh — a revoked grant takes effect instantly
Two queries on the hot path of every admin request
No cache-invalidation logic to get wrong
Load scales with traffic, not with the role table
Trivial to read and reason about
Redis sits idle while the DB absorbs the reads
Reads/req: 2 DB Staleness: 0 s Complexity: low

02Cache per-role permissions in Redisin use

Look each role_id up in the cache; on a miss, read the DB and back-fill with a TTL.

// permission.guard.ts:34-51 (current)
const keyOf = (id) => `role_user:${id}:perms`;
const keys  = roleIds.map((id) =>
  `role_per:${id}:perms`);
const cached = await Promise.all(
  keys.map((k) => cache.get(k)));

const perms = await Promise.all(roleIds.map(
  async (roleId, i) => {
    if (cached[i]) return cached[i];
    const p = await svc.getByRoleId(roleId);
    await cache.set(keyOf(roleId), p, TTL);
    return p;
  }));
Pro
Con
Warm path touches Redis only, not MariaDB
Write key ≠ read key — role_user: vs role_per:
Per-role keys are shared across all users of that role
No invalidation on grant change; waits for TTL
TTL bounds the staleness window
TTL is 36000 ms — the comment says "1h"
Reads/req: ~0 DB (warm) Staleness: ≤ 36 s Complexity: medium

03Bake permissions into the JWT

Embed the resolved permission codes as a claim at login; the guard reads them off the token.

// auth.service.ts — at sign-in
const perms =
  await resolvePermissions(user.id);
const accessToken = jwt.sign({
  sub: user.id.toString(),
  roles,
  perms,                  // new claim
});

// permission.guard.ts — zero I/O
if (!user.perms?.includes(required))
  throw new ForbiddenException();
Pro
Con
Authorization needs zero round-trips at all
Stale until the token expires or is rotated
Horizontally scales with no shared state
Fat tokens; sent on every request header
Pairs with the existing refresh rotation
Revoking a permission means revoking sessions
Reads/req: 0 Staleness: token life Complexity: medium

Recommendation

Keep approach 02, the Redis per-role cache — it is the right shape for an admin surface where many operators share a handful of roles, and per-role keys mean one cached entry serves every user of that role. But it needs two fixes before it can be trusted.

Bug to fix first. The back-fill writes with keyOf(roleId) → role_user:{id}:perms, but the read builds role_per:{id}:perms. The keys never match, so the cache never hits — every request silently falls through to the database, making this approach behave like approach 01 at approach 02's cost. Align both on one prefix, then make PERMISSION_CACHE_TTL match its "1h" comment (it is currently 36000 ms = 36 s), and drop the leftover console.log(userPerms).

Revisit approach 03 only once permission changes must propagate without a DB read at all — and only after wiring active invalidation, so a grant edit busts role_per:{id}:perms instead of waiting out the TTL.
