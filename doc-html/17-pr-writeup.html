Prompt Write up PR #134 for my reviewers. Explain why card-paid orders were going silent, walk them through the change file by file with the why for each, show before/after behavior, and point them at the idempotency edge. They know the COD path but not the webhook internals.
TL;DR

Only COD orders sent a confirmation email. The Sepay webhook marked an order PAID inside a transaction but never enqueued the email job, so every card-paying customer went silent. This PR enqueues send-order-email after the PAID transaction commits, reusing the existing outbox + BullMQ path, with a deterministic jobId so a re-delivered webhook can't double-send.

Why

When the Sepay integration landed, the COD branch in order.service.ts got the outbox+enqueue calls but the webhook success branch in hooks.service.ts did not. The order status flips to PAID correctly and stock is committed — the customer just never hears about it. With QR/transfer now the primary payment method, that's most of our confirmations going missing.

Before
  • COD order → outbox row + enqueueOrderEmail
  • Card/QR order → PAID, but no email
  • Customers ping support to confirm payment landed
  • No signal that the gap even exists
After
  • Both paths enqueue exactly one send-order-email
  • Enqueue happens only after the PAID tx commits
  • Re-delivered webhook is a no-op via jobId
  • Enqueue + dead-letter are logged for visibility

Where to focus your review

1
Enqueue strictly after commit
hooks.service.ts. The createOutboxOrderEmail + enqueueOrderEmail pair must run after handleSepayWebhookMarkDone resolves, never inside its transaction. Confirm I didn't accidentally pull it into the $transaction callback.
2
The idempotency chain
A duplicate webhook is stopped twice: first by the webhook_inbox DONE short-circuit, then by the deterministic jobId. Sanity-check that an order can only ever map to one outbox row, or the second guard leaks.
3
What I deliberately did not do
No itemized email body (still code/id/status), no email on SHORTFALL/EXPIRED, no backfill for historical PAID orders. Each layers on cleanly later; bundling them would make this unreviewable.
