How authentication flows through the codebase

The API uses stateless JWT access tokens with hashed, rotating refresh tokens. Two guards run globally on every request via APP_GUARD: AuthGuard verifies the bearer token and checks a revocation list, then PermissionsGuard resolves the caller's permission set. Raw tokens are never stored — only their SHA-256 hashes — so the revoked_token and user_session tables can be read without ever exposing a usable credential. That makes the guard pair the single trust boundary for the whole app.

src/common/security/guard/auth.guard.ts :24-57

The trust boundary. Routes flagged @Public() or @Refresh() bypass it. Otherwise it pulls the Bearer token, verifies the signature, hashes it, and asks RevokedTokenService whether that hash is on the deny-list — logout writes the hash there, so a logged-out token fails here even though its signature is still valid.

show source 
// auth.guard.ts
const isPublic  = reflector.getAllAndOverride(IS_PUBLIC_KEY, [...]);
const isRefresh = reflector.getAllAndOverride(IS_REFRESH_KEY, [...]);
if (isPublic || isRefresh) return true;

const token = this.extractTokenFromHeader(request);
if (!token) throw new UnauthorizedException();
try {
  const payload = await jwtService.verifyAsync(token);
  const hashedToken = tokenHash(token);
  if (await revokedTokenService.isRevoked(hashedToken))
    throw new UnauthorizedException(SecurityMessage.SESSION_REVOKED);
  request['user'] = payload;
} catch { throw new UnauthorizedException(); }

src/common/security/guard/permission.guard.ts :19-55

If the handler carries no @RequirePermissions(...) metadata, the guard returns early. Otherwise it loads the user's role IDs, then tries to read each role's permission list from Redis, falling back to the database on a miss and writing the result back with a TTL. The required code must be present in the flattened set or it throws ForbiddenException.

show source 
// permission.guard.ts
const roleIds = await userRoleService.getRoleIdsByUserId(user.sub);
const keyOf = (id) => `role_user:${id}:perms`;        // write key
const keys  = roleIds.map((id) => `role_per:${id}:perms`);  // read key
const cached = await Promise.all(keys.map((k) => cache.get(k)));

const permsByRole = await Promise.all(roleIds.map(async (id, i) => {
  if (cached[i]) return cached[i];
  const p = await rolePermissionService.getByRoleId(id);
  await cache.set(keyOf(id), p, PERMISSION_CACHE_TTL);
  return p;
}));
if (!new Set(permsByRole.flat()).has(required))
  throw new ForbiddenException();

src/utils/hashToken.util.ts :1-6

Both the access-token deny-list and the refresh-token sessions store hashes, not raw tokens. tokenHash is just an alias of hashToken — a single SHA-256 over the token. A database dump therefore leaks no usable credentials.

show source 
// hashToken.util.ts
export const hashToken = (token: string) =>
  createHash('sha256').update(token).digest('hex');

export const tokenHash = hashToken;

src/modules/auth/auth.service.ts · refreshToken()

On refresh, the RefreshGuard validates the hashed refresh token against an ACTIVE, non-expired session and attaches it. The service then re-reads the user's roles from the DB, signs a fresh token pair, and rotates the session — so a refreshed access token always reflects the current role set, even if the cached permissions lag behind.

Key files

  • security/guard/auth.guard.ts Verifies JWT + checks the revocation deny-list.
  • security/guard/permission.guard.ts Resolves per-role permissions, Redis-cached.
  • security/security.module.ts Wires both guards globally as APP_GUARD.
  • modules/auth/auth.service.ts Login, refresh rotation, logout/revoke.
  • modules/revoked-token/* Hash deny-list for logged-out access tokens.
  • utils/hashToken.util.ts SHA-256 used for every stored token.

Gotchas

  • The permission cache write key (role_user:) does not match the read key (role_per:), so the cache never hits and every guarded request falls through to the DB.
  • PERMISSION_CACHE_TTL is 36000 ms (36 s), although the inline comment says "1h". Permission changes can lag up to that window once the keys are fixed.
  • A leftover console.log(userPerms) sits on the hot path in permission.guard.ts.
  • Revocation is enforced for access tokens only via hash lookup; there is no jti claim, so tokens are revoked by hash, not by id.
