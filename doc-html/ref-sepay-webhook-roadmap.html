Sepay Webhook Integration Roadmap (NestJS + Prisma)

Tài liệu này là lộ trình + code mẫu để bạn tự triển khai dần. Không phải full source code đã cắm sẵn vào project.

#1) Mục tiêu phase 1

  1. POST /api/v1/payments/sepay/checkout: tạo đơn chờ thanh toán.
  2. User chuyển khoản theo nội dung chuẩn (prefix + orderCode).
  3. POST /api/v1/payments/sepay/webhook/:secret: nhận webhook Sepay.
  4. Auto cập nhật orders + log vào payment_transactions + webhook_inbox.

#2) Ý tưởng tổng quát

  1. Checkout tạo order ở trạng thái PENDING_PAYMENT.
  2. Backend trả transferContent và qrUrl để frontend hiển thị.
  3. Sepay gửi webhook khi có giao dịch vào.
  4. Backend dùng code (hoặc fallback từ content) để tìm order.
  5. Nếu khớp tiền: chuyển order sang PAID.
  6. Nếu không khớp: ghi FAILED, giữ order pending.

#7) Service logic mẫu

File: src/modules/payment/payment.service.ts

Luồng createCheckout:

  1. Lấy cart mới nhất của user + items.
  2. Validate cart không rỗng.
  3. Tính subtotal/totalAmount.
  4. Tạo order (PENDING_PAYMENT, PENDING).
  5. Tạo payment_transactions trạng thái PENDING với gateway SEPAY.
  6. Sinh transferContent = PREFIX + orderCode.
  7. Trả qrUrl, amount, transferContent.

Luồng handleWebhook:

  1. Verify secret.
  2. Insert webhook_inbox PENDING theo providerEventId = payload.id.
  3. Nếu duplicate unique key: return { accepted: true, duplicate: true }.
  4. Nếu transferType !== 'in': mark DONE và return.
  5. Parse orderCode từ code hoặc content.
  6. Tìm order.
  7. So khớp amount.
  8. Nếu match: tạo payment SUCCESS, update order PAID/SUCCESS, mark inbox DONE.
  9. Nếu mismatch: tạo payment FAILED, mark inbox FAILED.

#10) Checklist test thủ công

  1. Checkout trả transferContent + qrUrl đúng.
  2. Webhook hợp lệ đổi order sang PAID.
  3. Gửi lại cùng id webhook không tạo record success lần 2.
  4. code = null nhưng content có mã đơn vẫn match.
  5. Amount lệch thì order không thành công.
  6. Secret sai thì reject.
  7. transferType = out được ignore.

#11) Lưu ý triển khai

  1. Không tin payload.gateway là enum hệ thống, lưu raw vào JSON payload để đối soát.
  2. webhook phải trả ACK nhanh, không xử lý nặng blocking lâu.
  3. Toàn bộ cập nhật paid nên đi trong transaction DB.
  4. Có thể bổ sung cron reconcile ở phase 2 (không làm trong phase này).
