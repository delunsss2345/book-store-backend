01

Deterministic Recommendation Engine (No LLM)

Đây là tài sản kỹ thuật mạnh nhất của dự án: một bộ gợi ý deterministic, giải thích được, không phụ thuộc runtime của model ML — rẻ, nhanh, testable, và reproducible.

▸Weighted Scoring + Time Decay

Mỗi hành vi người dùng được quy về một điểm ý định = trọng số sự kiện × hệ số suy giảm theo thời gian. Mua thật (PAYMENT_SUCCESS) nặng gấp 12 lần một lượt xem.

Sự kiệnTrọng số
PAYMENT_SUCCESS+12
PLACE_ORDER+8
CHECKOUT_START+6
ADD_TO_CART+4
VIEW_BOOK+1
REMOVE_FROM_CART−2
PAYMENT_FAILED−1
src/common/constants/event-type.constant.ts

▸Core Formula

Tín hiệu âm (gỡ khỏi giỏ, thanh toán lỗi) trừ điểm — điều mà collaborative filtering ngây thơ thường bỏ qua.

// suy giảm theo số ngày: ưu tiên hành vi gần đây
function timeDecay(daysAgo) {
  if (daysAgo <= 3)  return 1.0;  // 100%
  if (daysAgo <= 14) return 0.7;
  if (daysAgo <= 30) return 0.4;
  return 0.2;                  // cũ → 20%
}

score = EVENT_WEIGHT[type] * timeDecay(daysAgo);
Điểm phỏng vấn: vì sao chọn deterministic thay vì ML? → Cold-start tốt, giải thích được cho business, không cần GPU/training pipeline, kết quả ổn định để viết unit test. Đây là một quyết định kiến trúc có lý do, không phải hạn chế.

Pipeline 3 luồng → hợp nhất → top-K

Variant events
view/cart/pay + Order events
theo đơn hàng → Co-purchase graph
order_items → merge + boost theo qty → getTopKMap(20)

Luồng đơn hàng được mở rộng bằng đồ thị mua-kèm: từ order_id → order_items → snapshot → variant, khám phá sách thường được mua cùng nhau (novelty), rồi hợp nhất vào điểm variant với hệ số nhân theo số lượng. Hàm upsertScore() giữ lại timestamp gần nhất để decay luôn chính xác.

▸Related Books via Jaccard

Gợi ý "sách tương tự" dùng Jaccard Index (Intersection over Union) trên tập category — đơn giản, đối xứng, không cần embedding.

Semantic Search: Gemini Embeddings + Pinecone

Search không phải LIKE '%keyword%'. Người dùng gõ ý tưởng tự nhiên → vector 768 chiều → đối sánh độ tương đồng trong Pinecone.

Query "sách self-help nhẹ nhàng" → Gemini embed · 768D → Pinecone topK → giữ thứ hạng vector → cache Redis

▸Batch Indexing

Reindex toàn bộ catalog bằng batch embedding (200 record/lần) để tránh rate-limit, kèm validate giá & bỏ qua record lỗi thay vì fail cả lô.

  • Metadata enrichment: categoryIds, langHints, price
  • Namespace book-store, upsert idempotent theo id
  • Endpoint reindex được bảo vệ bằng quyền search:reindex_books
src/modules/pinecone/pinecone.service.ts

▸Rank Preservation + Cache

Pinecone trả về thứ tự liên quan; service map bookId → rank rồi sort lại dữ liệu DB theo rank đó để không mất relevance khi hydrate chi tiết sách.

// key cache phản ánh đủ tham số truy vấn
`query:books-sematic:${q}:${langId}:${page}:${limit}`

Auth: JWT + Refresh, Session, Device, Brute-Force Defense

Không chỉ "login trả token" — đây là một hệ thống danh tính nhiều lớp với khả năng thu hồi tức thì.

▸Token pair

Access JWT ngắn hạn + refresh token opaque lưu hash. Logout thu hồi song song cả access (blacklist tới exp) lẫn session.

auth.service.ts · revoked-token

▸Revocation Check in Guard

AuthGuard verify chữ ký rồi tra isRevoked(tokenHash) — token bị logout không còn dùng được dù chưa hết hạn.

common/security/guard/auth.guard.ts

▸Device fingerprinting

Upsert thiết bị theo (userId, fingerprint), gắn session với device → nền tảng cho "thiết bị tin cậy".

user-device · user-session

▸Brute-force backoff

Resend OTP: tối đa 5 lần/24h, chặn 5h nếu vượt ngưỡng; mọi lần đăng nhập (kể cả thất bại) đều ghi login_attempt.

login-attempt · email-outbox

▸Email Verify via Queue

Mã xác thực được tạo + đẩy vào BullMQ; cũ bị vô hiệu trước khi sinh mã mới (chống đua & spam).

Cached RBAC: Authorization Without DB Round-Trips

Permission được kiểm ở tầng guard, nhưng được cache theo role trong Redis để không truy vấn DB mỗi request.

▸Decorator → Guard → Cache

Controller khai báo @RequirePermissions(...); guard đọc role của user, nạp quyền song song, ưu tiên cache role_per:{id}:perms (TTL 1h), gộp thành Set rồi kiểm tra.

// nạp quyền theo từng role, cache-first, chạy song song
const keys = roleIds.map(id => `role_per:${id}:perms`);
const perms = await Promise.all(roleIds.map(async (id,i) =>
  cached[i] ?? fetchAndCache(id)));
if (!new Set(perms.flat()).has(required))
  throw new ForbiddenException();
common/security/guard/permission.guard.ts

▸Normalized Data Model

Quan hệ nhiều-nhiều đầy đủ: User → UserRole → Role → RolePermission → Permission, dùng composite unique key chống cấp quyền trùng.

UserUserRoleRole RolePermissionPermission

Permission là enum có chủ đích (admin:create-book, role_permission:grant…) → tự-tài-liệu & type-safe.

modules/permission · role · role-permission · user-role
Vấn đề mời gọi cải tiến: cache theo role rất nhanh, nhưng cần invalidation khi grant/revoke quyền — đây chính là phần đáng "đào sâu" ở mục cuối để tăng giá trị CV (xem §Cần đào sâu thêm).

Payment Webhook & Outbox: Where Errors Aren't Allowed

Tiền bạc đòi hỏi idempotency và nhất quán giao dịch — dự án xử lý cả hai một cách bài bản.

▸Idempotent Sepay Webhook

Mỗi sự kiện lưu vào webhook_inbox với composite unique (gateway, providerEventId) bằng upsert → webhook bắn lại nhiều lần cũng chỉ xử lý một lần.

  • Cập nhật order + payment trong một transaction
  • order.status = PAID & paymentStatus = SUCCESS nguyên tử
  • Lưu raw payload để truy vết/đối soát
modules/hooks/hooks.repository.ts

▸Outbox Pattern for Email

Email không gửi inline. Bản ghi vào email_outbox (PENDING) → đẩy BullMQ với jobId idempotent + exponential backoff (3s→9s→27s, 3 lần).

add('send-email', payload, {
  attempts: 3,
  backoff: { type: 'exponential', delay: 3000 },
  jobId: `email-${outboxId}`, // idempotent
  removeOnComplete: true,
});
modules/jobs/producers/email.producer.ts
Why outbox: tách "ghi nhận ý định gửi" khỏi "gửi thật" → trạng thái PENDING→SENDING→SENT/FAILED theo dõi được, retry an toàn, không mất mail khi SMTP chập chờn. Đây là pattern enterprise mà nhà tuyển dụng đánh giá cao.

Going Deeper to Make the Project Shine

Mỗi điểm dưới đây vừa nâng chất lượng kỹ thuật, vừa cho bạn câu chuyện hay khi phỏng vấn.

1Measuring Recommendation Quality

Thêm metric offline (CTR giả lập, hit-rate@K, coverage) + A/B hook. Hiện chưa có cách chứng minh recommendation tốt hơn baseline.

2Permission Cache Invalidation

Khi grant/revoke, chủ động xoá role_per:{id}:perms. Tránh "quyền cũ sống thêm 1 giờ" — một lỗ hổng bảo mật tinh tế.

3Refresh token rotation + reuse detection

Xoay refresh mỗi lần dùng, phát hiện tái sử dụng token đã xoay → thu hồi cả họ session. Nâng auth lên chuẩn ngân hàng.

4Testing & Observability

Unit test cho scoreEvent/Jaccard (đang deterministic → dễ test), structured logging, tracing webhook → tăng độ tin cậy.

5Webhook Signature Verification

Idempotency đã tốt; bổ sung xác thực chữ ký/secret từ Sepay để chống giả mạo callback thanh toán.

6Hybrid Reranking

Kết hợp điểm hành vi (§01) với điểm ngữ nghĩa (§02) thành một bộ rerank duy nhất — câu chuyện "personalized semantic search" rất ấn tượng.
