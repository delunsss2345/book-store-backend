#Task: Catalog categories endpoint with cache-first tree response

#Task

  • Implement GET /catalog/categories.
  • Return only active categories and build recursive children.
  • Support lang in vi|en with default en.
  • Use cache-first flow before querying database.

#Endpoint

  • GET /api/v1/catalog/categories?lang=en|vi

#Input

  • Query param:
  • lang (optional): vi or en, default en.

#Process Flow

  1. Read lang from query, normalize to lowercase, default to en.
  2. Validate lang is only vi or en. Invalid value returns 400.
  3. Build category cache key: catalog:categories:{lang}.
  4. Check cache:
  • If hit: return cached tree immediately.
  • If miss: continue.
  1. Resolve language id from languages table using active language code.
  2. Query active categories from categories with translation from category_translations by languageId.
  3. Build tree:
  • Map rows by id.
  • Attach child node to parent node.
  • Keep only root nodes (parentId = null).
  • Skip rows without translation.
  • Skip orphan rows if parent is not present in filtered set.
  1. Sort nodes by sortOrder, then id.
  2. Save tree to cache with TTL 60000.
  3. Return tree response.

#DB Tables

  • languages
  • categories
  • category_translations

#Cache Keys + TTL

  • Language cache: catalog:lang:{lang} with TTL 60000.
  • Category tree cache: catalog:categories:{lang} with TTL 60000.

#Task: Catalog books list with language filter and cheapest variant

#Task

  • Implement simple list endpoint GET /catalog/books.
  • Join books with book_translations.
  • Filter translation by language code only (vi or en).
  • Return only cheapest active variant price for each book.
  • If a book has no active variant, mark product as out of stock.
  • Support pagination and cache by page/limit/filter.

#Endpoint

  • GET /api/v1/catalog/books?page=1&limit=20&lang=en

#Input

  • Query params:
  • page (optional, default 1)
  • limit (optional, default 20)
  • lang (optional, vi|en, default en)

#Process Flow

  1. Read page, limit, lang.
  2. Resolve language by lang (default en).
  3. Build cache key:
  • catalog:books:lang={lang}:p{page}:l{limit}
  1. Check cache:
  • If hit: return cached payload.
  • If miss: continue.
  1. Build DB filter:
  • Base: active book + not deleted.
  • Translation condition: book_translations.language_id = languageId
  1. Query total count.
  2. Query paginated books with:
  • translation row (title, slug)
  • only 1 active variant with min price (order by price asc, id asc, take 1)
  1. Map response item:
  • minPrice = cheapest variant price
  • isOutOfStock = true when no variant or stock <= 0
  1. Save to cache and return { page, limit, total, totalPages, items }.

  • If token is invalid: create guest_sessions record and mark request.authError = true.

#Task: POST /cart/items with guest-db + user-db flow (with cleanup)

#Task

  • Input:
  • bookVariantId (required number).
  • Every request increases by +1.

#Endpoint

  • POST /api/v1/cart/items

#Process Flow

  1. Controller reads request context from cart guard (user, authError, guestSessionId) and DTO body.
  2. Service normalizes bookVariantId to bigint and decides branch:
  • Guest branch when authError = true or request has no user id.
  • User branch when user id exists and auth is valid.
  1. Guest branch:
  • Read guest cart by guestSessionId (findByGuestSessionId).
  • If guest cart not found: create cart by guestSessionId, then create cart_item with quantity 1.
  • If guest cart exists and same variant exists: increase quantity by +1 in database.
  • If guest cart exists and variant does not exist: create cart_item with quantity 1.
  • Return payload with authError = true and cart item data.
  1. User branch:
  • Verify user exists via authRepository.findUserById(userId).
  • Find cart by userId; create cart if missing.
  • Find cart item by (cartId, bookVariantId):
  • If exists: update quantity = current + 1 in DB.
  • If not exists: create new item with quantity = 1 in DB.
  1. Cleanup strategy for guest data:
  • Guest cart data is persisted to support smoother guest experience across requests.
  • Old guest carts and related items must be removed by scheduled cleanup (TTL/time-based deletion) to avoid data growth.

  • Guest:
  • User:

#Task: Cart quantity-delta API and remove direct patch API

#Task

  • Remove direct overwrite API: PATCH /api/v1/cart/items/:itemKey.
  • Use only delta API: PATCH /api/v1/cart/items/:itemKey/delta.
  • Support custom increase/decrease by quantity delta.
  • Keep delete APIs:
  • DELETE /api/v1/cart/items/:itemKey to remove one cart item.
  • DELETE /api/v1/cart to remove cart and all related cart items.
  • Keep POST /api/v1/cart/merge as pending (not implemented now).

#Endpoint Contract

  • PATCH /api/v1/cart/items/:itemKey/delta
  • Request body:

#Process Flow

  1. FE may batch fast clicks (increase/decrease) and send final delta once timeout is reached.
  2. BE resolves actor from request context:
  • guest (authError = true)
  • user (authError = false)
  1. Find target item by itemKey and actor scope:
  • guest: item must belong to guest cart (guestSessionId)
  • user: item must belong to user cart (userId)
  1. Calculate new quantity:
  • newQuantity = currentQuantity + delta
  • Clamp lower bound: if below 0 -> set to 0
  • Clamp upper bound by variant stock:
  • if requested quantity exceeds stock -> set to stock
  1. Save updated quantity to DB.
  2. Return actor-specific response with updated item info.

#Notes

  • Cart quantity is behavioral data, not a guaranteed purchase intent.
  • High quantity attempts are capped by stock.
  • Very large negative delta will clamp to 0 instead of negative values.

#Task: Review routes workflow (AI draft + create review + list review)

#Task

  • Add AI-assisted draft route for review text generation.
  • Add guarded create-review route that only allows purchased variants.
  • Keep public list-review-by-book-slug route.
  • Reuse review business rule for both AI draft and create:
  • user must have purchased the variant
  • user can review only once for the same userId + bookId + bookVariantId.

#Endpoints

  • POST /api/v1/reviews/review/draft (auth required)
  • POST /api/v1/reviews (auth required)
  • GET /api/v1/reviews/books/:slug?lang=vi|en&page=1&limit=20 (public)

#Input Contract

  1. POST /reviews/review/draft
  • body:
  • bookId (number)
  • bookVariantId (number)
  • userHint (string)
  • tone? (string)
  • language? (vi|en)
  • targetWords? (number)
  1. POST /reviews
  • body:
  • bookId (number)
  • bookVariantId (number)
  • rating (1..5)
  • content? (string)
  1. GET /reviews/books/:slug
  • path:
  • slug (required)
  • query:
  • lang? (vi|en)
  • page? (default 1)
  • limit? (default 20)

#Process Flow

#A) AI draft flow - POST /reviews/review/draft

  1. Controller extracts user.sub from access token and parses to bigint.
  2. Service converts bookId, bookVariantId to bigint.
  3. Check purchase permission:
  • query order_items -> book_variant_snapshots -> orders
  • order status must be in: PAID, CONFIRMED, PACKING, SHIPPING, DELIVERED.
  1. Check duplicate review in reviews with composite logic:
  • same userId + bookId + bookVariantId.
  1. Validate userHint (trim + not empty).
  2. Load book detail from catalog service by bookId + language (default vi).
  3. Validate bookVariantId exists inside loaded book variants.
  4. Call Gemini service with context:
  • title, format, userHint, targetWords.
  1. Return draft response:
  • bookId, bookVariantId, draftText, wordCount.

#B) Create review flow - POST /reviews

  1. Controller extracts user.sub and parses to bigint.
  2. Service parses bookId, bookVariantId from request body.
  3. Re-check purchased variant rule (same query logic as AI draft).
  4. Re-check duplicate review (userId + bookId + bookVariantId).
  5. Normalize content (trim, optional nullable).
  6. Create review row in reviews.
  7. Return created review payload:
  • reviewId, userId, rating, content, createdAt, variant.

#C) List review flow - GET /reviews/books/:slug

  1. Validate slug not empty.
  2. Resolve language by lang (en default in current implementation).
  3. Find book by translation slug.
  4. Query review count + paginated rows by bookId.
  5. Map response:
  • page, limit, total, totalPages, items.

#Task: Hyper recommend merge 2 event streams (variant + order)

#Task

  • Document the main recommendation algorithm as a sequence-ready workflow.
  • Clarify how 2 streams are merged:
  • stream A: bookVariant interaction events.
  • stream B: order transaction events.
  • Keep this workflow aligned with current code in user-event.service.ts.

#Endpoint

  • GET /api/v1/user-events/hyper-recommend/books

#Business Goal

  • Personalize recommendation feed based on real user behavior and transaction intent.
  • Convert user interest into purchase-supporting suggestions ("co-buy") instead of generic bestseller-only list.
  • Keep explainable and deterministic logic (no LLM call at request time).

#Business Scope

  • In scope:

  • logged-in user only (JWT required).

  • recommendation from user event streams + order-item co-buy graph.

  • return book-facing payload for FE recommendation block.

  • Out of scope:

  • guest recommendation persistence.

  • ML model training / batch ranking table.

  • explicit campaign boost or merchandising override.

#Business Workflow (Domain View)

  1. User opens recommendation entrypoint (hyper-recommend).
  2. System reads two intent sources:
  • interaction intent (bookVariant events): view/cart behavior.
  • transaction intent (order events): checkout/payment behavior.
  1. System transforms both sources into comparable scores with time decay.
  2. System merges both sources into one unified bookVariant preference space.
  3. System expands preference set via co-buy relation from real orders.
  4. System resolves candidates into active catalog entities.
  5. System returns recommendation list or empty list when signal is insufficient.

#Business Rules

  • Rule 1: only valid event families are scored.

  • bookVariant: VIEW_BOOK, ADD_TO_CART, REMOVE_FROM_CART.

  • order: CHECKOUT_START, PLACE_ORDER, PAYMENT_SUCCESS, PAYMENT_FAILED.

  • Rule 2: all merged scores must be normalized to bookVariantId key-space.

  • Rule 3: if stream B (order) exists, it can amplify stream A by:

  • orderId -> orderItem -> snapshotId -> variantId.

  • quantity factor from orderItem.quantity.

  • Rule 4: co-buy generation must exclude seed snapshots to avoid echoing input items.

  • Rule 5: fail-fast behavior is allowed.

  • if any critical stage has empty data, return [].

  • Rule 6: output is deterministic for same input snapshot and data state.

#Business Inputs and Semantics

  • objectType = "bookVariant":

  • means "user interacted with a specific sellable variant".

  • objectType = "order":

  • means "user reached transactional milestone on a concrete order".

  • objectId:

  • stored as string but semantically must map to existing BigInt IDs.

#Business Output Contract

  • Returns recommendation list tailored for FE block.
  • Each item is resolved from candidate bookVariantId into book-facing payload.
  • If business signal quality is low, returns [] (no noisy fallback in this flow).

#Business Acceptance Criteria

  • User with only interaction events receives non-empty recommendations when co-buy graph exists.
  • User with only order events still receives recommendations via conversion to variant preference space.
  • User with mixed streams receives merged behavior (not overridden by one stream only).
  • User without valid events receives [].
  • Response must remain deterministic and explainable from stored events.

#Input Contract

  • Auth:

  • required JWT, userId = BigInt(user.sub).

  • Event stream A (bookVariant):

  • objectType = "bookVariant"

  • objectId = <bookVariantId string>

  • allowed score events: VIEW_BOOK, ADD_TO_CART, REMOVE_FROM_CART

  • Event stream B (order):

  • objectType = "order"

  • objectId = <orderId string>

  • allowed score events: CHECKOUT_START, PLACE_ORDER, PAYMENT_SUCCESS, PAYMENT_FAILED

#Process Flow (Step-by-step)

  1. Controller receives request and resolves userId.
  2. Service computes 2 score maps in parallel:
  • map A from objectType = bookVariant.
  • map B from objectType = order.
  1. Service merges map A + map B into one variant map:
  • convert orderId -> orderItems -> snapshotIds -> variantIds.
  • add order score into matching variant score.
  • apply quantity factor from orderItem.quantity.
  1. Service converts merged variant ids into seed snapshot ids.
  2. Service gets all order ids containing those seed snapshots.
  3. Service groups "also bought" candidate snapshots from those orders, excluding seed snapshots.
  4. Service maps candidate snapshot ids back to candidate variant ids.
  5. Service dedupes candidate variant ids and loads catalog rows by variant ids.
  6. Service maps rows to response payload and returns result.
  7. If data is empty at any stage, return [] early.

#Score + Ranking Notes

  • Event score uses scoreEvent(...) (weight + time decay).
  • Merge keeps data in one key-space (bookVariantId) to simplify candidate generation.
  • Positive-score filter is applied before top-K trimming.
  • Tie-break rule: newer createdAt wins when scores are equal.
